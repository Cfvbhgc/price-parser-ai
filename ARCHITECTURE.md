# Архитектура простыми словами

Здесь объяснено, **как файл проходит путь от кривого прайса до аккуратного
отчёта** — без сложных терминов. Если README — «что запустить», то здесь —
«что происходит внутри и почему так».

---

## Главная проблема одним абзацем

Поставщики присылают прайсы в Excel, и у каждого — своя каша: сверху название
фирмы и телефоны, потом пустые строки, колонки называются по-разному и стоят в
разном порядке, часть ячеек объединена. Человек глазами разберётся за минуту, а
жёсткая программа («бери цену из колонки C, строку 5») сломается на первом же
непохожем файле. Наша задача — сделать программу, которая **разбирается сама**,
как человек: находит, где шапка, понимает, где какая колонка, и достаёт товары.

Аналогия: это как разобрать стопку разных чеков из разных магазинов. Формы
разные, но на каждом есть «название», «цена», «количество» — надо просто их
узнать, а не запоминать точное место.

## Путь файла по шагам

```
   [1] прайс .xlsx / .csv
        │
        ▼
   [2] reader.py            превращаем в простую таблицу (сетку),
                            разворачиваем объединённые ячейки
        │
        ▼
   [3] header_detector.py   находим строку-ШАПКУ, мусор сверху игнорируем
        │
        ▼
   [4] structure.py         определяем, ГДЕ КАКАЯ КОЛОНКА:
        │                     • LLM по образцу  (основной путь)
        │                     • эвристика       (запасной, без ключа)
        ▼
   [5] extractor.py         по найденной разметке разбираем ВЕСЬ файл кодом,
                            превращаем «1 950,00» в число 1950, чистим мусор
        │
        ▼
   [6] categorizer.py       каждому товару ставим категорию по ключевым словам
        │
        ▼
   [7] report.py            рисуем красивый Excel: тёмная шапка, зебра, сетка
```

### Шаг 2. Чтение и объединённые ячейки

Excel читаем в простую таблицу «строки × ячейки». Главная хитрость —
**объединённые ячейки**: когда в Excel объединяют несколько клеток в одну,
значение хранится только в левой верхней, а остальные пустые. Если это не
учесть, шапки и подзаголовки «разъезжаются». Поэтому мы копируем значение во все
клетки объединённого диапазона — таблица становится ровной.

CSV читаем с автоопределением разделителя: запятая, точка с запятой или таб —
у поставщиков встречается всё.

### Шаг 3. Где шапка? (отсечение мусора)

Мы **не считаем, что шапка — первая строка**. Вместо этого просматриваем верхние
строки и для каждой считаем: сколько разных «ролей» колонок в ней угадывается по
ключевым словам (название, артикул, цена, количество). Строка «ООО КрепёжТорг»
даст 0 совпадений, а строка «Наименование | Артикул | Цена | Остаток» — целых 4.
Побеждает строка с максимумом — это и есть шапка. Всё, что выше, — мусор,
который мы отбрасываем.

### Шаг 4. Где какая колонка? И почему LLM зовётся ОДИН раз

Это ключевой шаг. Нужно понять: в этом файле «название» — вторая колонка, «цена»
— четвёртая и т.д. Здесь помогает нейросеть, но **с важной оговоркой про экономию**.

**Почему нельзя слать в LLM весь файл.** Прайс может быть на тысячи строк. Гонять
каждую строку через нейросеть — это:
- дорого (платят за объём текста — «токены»);
- медленно (тысячи запросов);
- незачем — структура-то одна на весь файл.

**Что мы делаем вместо этого.** Отправляем в модель только **образец**: строку
заголовка и 5 строк данных под ней. Этого достаточно, чтобы понять структуру.
Модель возвращает компактный ответ — «карту» колонок в JSON:

```json
{"name": 1, "article": 0, "price": 3, "quantity": 2}
```

Это значит: название — колонка 1, артикул — 0, цена — 3, количество — 2.
**Дальше нейросеть не нужна вообще** — весь файл (хоть 10 000 строк) разбирает
обычный код по этой карте. Один вызов LLM на файл вместо тысяч.

Аналогия: мы не просим переводчика читать всю книгу вслух. Показываем ему первую
страницу, он говорит «глава называется так, абзацы устроены так» — и дальше мы
читаем сами по его подсказке.

**Запасной вариант без нейросети.** Если ключа API нет или модель не ответила,
структуру определяет **эвристика**: тот же поиск по ключевым словам, что и для
шапки, но теперь — какая колонка какую роль играет. На типовых прайсах она
справляется, поэтому инструмент работает и полностью офлайн. В отчёте всегда
видно, каким способом определена структура.

### Шаг 5. Разбор всего файла и цены

Пройдя по всем строкам под шапкой, берём ячейки по «карте» колонок. Попутно:
- **парсим цену**: «1 950,00», «12 990 ₽», «72,50» → числа `1950`, `12990`,
  `72.5` (убираем пробелы-разделители тысяч, валюту, приводим запятую к точке);
- **чистим мусор**: строки без названия пропускаем; строки-разделители вроде
  «=== САНТЕХНИКА ===» (есть текст, но нет ни цены, ни артикула) — не товар,
  тоже пропускаем.

### Шаг 6. Категории

У каждой категории есть список ключевых слов (`categories.json`). Смотрим на
название товара и относим его к первой подходящей категории. Тонкость: слова
могут быть подстроками друг друга — «шуруп» (крепёж) сидит внутри «шуруповёрт»
(инструмент). Поэтому при совпадении выигрывает **более длинное, более
специфичное** слово: «шуруповёрт» победит «шуруп», и товар уйдёт в «Инструмент».
Список категорий и слов легко расширяется под нужную отрасль без правки кода.

### Шаг 7. Красивый отчёт

Собираем результат в Excel через openpyxl: заголовок отчёта, тёмная шапка с белым
жирным шрифтом, чередование строк (зебра) для читаемости, тонкие границы,
включённая сетка, автоширина колонок, автофильтр и закреплённая шапка. Получается
файл, который не стыдно отдать заказчику или загрузить в 1С.

## Почему так, а не проще

- **Почему не «цена всегда в колонке C»?** Потому что у следующего поставщика она
  в колонке E. Мы определяем структуру, а не зашиваем её.
- **Почему LLM, а не только ключевые слова?** Ключевые слова покрывают типовые
  случаи, но заголовки бывают нестандартные («Цена розн. с НДС», «Ном. позиции»).
  Нейросеть понимает смысл. При этом мы зовём её экономно — один раз на образце.
- **Почему LLM на структуру, а не на категоризацию каждой строки?** Категоризация
  по ключевым словам быстрая, бесплатная и предсказуемая. Гонять каждую позицию
  через нейросеть было бы дорого и медленно без реальной пользы.
- **Почему есть работа без ключа?** Чтобы инструмент работал офлайн и его можно
  было показать/протестировать без затрат, а нейросеть подключать по желанию.

## Коротко, чтобы пересказать заказчику

> «Инструмент берёт любой прайс поставщика — даже кривой, с мусором сверху и
> разными колонками — и сам разбирается в его структуре: находит шапку,
> понимает, где название, артикул, цена. Чтобы это было дёшево, нейросети
> показывают только маленький образец файла, она один раз говорит "вот карта
> колонок", а дальше весь файл — хоть на тысячи строк — разбирает обычный код по
> этой карте. Потом товары раскладываются по категориям и выгружаются в
> аккуратный Excel с оформлением. Работает и без нейросети — по ключевым словам.»
